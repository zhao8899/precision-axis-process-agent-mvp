# API与数据模型期望

本文定义本地MVP控制台的接口和数据期望，供应用实现对齐。命名可按技术栈调整，但行为不得偏离 `MVP开发交付包/`。

## 设计原则

- 知识库内容以交付包文件为准，应用只做读取、展示、检索和传给模型的上下文组织。
- 测试必须调用真实模型或真实配置的本地模型服务，不得返回预写标准答案。
- 系统只记录“模型回答、人工判定、证据备注”，不得自动判定合格放行。
- 遇到缺少实测值、企业证明、学校确认值时，输出应提示需确认，不得补造数据。

## 核心接口期望

| 接口能力 | 输入 | 输出 | 必须满足 |
|---|---|---|---|
| 健康检查 | 无 | 服务状态、知识库加载状态、模型配置状态 | Docker启动后可用于 smoke test |
| 知识库列表 | 无 | KB文件名、标题、版本/更新时间、加载状态 | 至少覆盖KB-01至KB-04和合并知识库 |
| 知识库内容读取 | 文件标识 | Markdown正文或结构化片段 | 只读，不改写交付包内容 |
| 问答请求 | 用户问题、可选场景、会话ID | 模型回答、引用的KB片段、边界提醒 | 应包含系统提示词和相关KB上下文 |
| G1-G8规则查询 | G编号或全部 | 名称、提醒内容、依据、处置、输出物、禁止表述 | 来源为KB-02 |
| 测试用例列表 | 无 | T01-T08编号、问题、期望要点、通过标准 | 来源为KB-04 |
| 测试执行 | 测试编号、模型配置、执行人 | 真实回答、时间、原始输出摘要 | 不得使用标准判定稿替代模型回答 |
| 人工判定保存 | 测试运行ID、通过/不通过、判定说明 | 已保存记录 | 判定依据为标准判定稿 |
| 验收汇总 | 测试批次ID | T01-T08结果、失败项、复测建议 | 失败项按缺陷修正规则处理 |

## 数据模型期望

### KnowledgeSource

| 字段 | 说明 |
|---|---|
| `id` | 稳定文件标识 |
| `fileName` | 交付包文件名 |
| `title` | 文档标题 |
| `category` | system_prompt、kb、test_case、acceptance、demo、version |
| `content` | Markdown原文或解析后的正文 |
| `sourcePath` | 本地相对路径 |
| `loadedAt` | 加载时间 |

### ChatSession

| 字段 | 说明 |
|---|---|
| `id` | 会话标识 |
| `scenario` | process、parameter、quality、exception、archive、speech、value |
| `createdAt` | 创建时间 |
| `modelProvider` | 本地配置的模型供应商名称或占位标识 |
| `knowledgeVersion` | 知识库版本或加载批次 |

### ChatMessage

| 字段 | 说明 |
|---|---|
| `id` | 消息标识 |
| `sessionId` | 所属会话 |
| `role` | user、assistant、system |
| `content` | 消息正文 |
| `referencedSources` | 使用的交付包来源 |
| `boundaryWarnings` | 触发的合规边界提醒 |
| `createdAt` | 创建时间 |

### TestCase

| 字段 | 说明 |
|---|---|
| `id` | T01至T08 |
| `question` | 测试问题 |
| `expectedPoints` | 期望回答要点 |
| `passCriteria` | 通过标准 |
| `source` | KB-04和标准判定稿 |

### TestRun

| 字段 | 说明 |
|---|---|
| `id` | 测试批次标识 |
| `caseId` | T01至T08 |
| `promptSent` | 实际发送的问题 |
| `modelAnswer` | 真实模型回答 |
| `answerSummary` | 人工或系统摘要 |
| `manualResult` | pass、fail、pending |
| `manualReason` | 判定说明 |
| `tester` | 测试人 |
| `createdAt` | 执行时间 |

### AcceptanceSummary

| 字段 | 说明 |
|---|---|
| `id` | 汇总标识 |
| `testRunIds` | 纳入汇总的测试记录 |
| `allPassed` | 是否全部人工判定通过 |
| `failedCases` | 未通过或待复测编号 |
| `exportText` | 可回填验收表的摘要 |
| `createdAt` | 汇总时间 |

## 固定边界校验点

应用层可做关键词风险提示，但最终回答仍以系统提示词和知识库约束为准。

| 风险输入 | 期望处理 |
|---|---|
| “自主开发AI” | 提醒正确表述为基于学校授权平台配置知识库、提示词、质量规则与测试用例 |
| “AI自动判断合格” | 明确不能自动放行，放行由1号质检员依据实测数据作出 |
| “企业真实订单/利润” | 要求企业证明或确认，不得编造 |
| “跳过首件检验” | 明确不可以，必须首件检验后进入下一工步 |
| 未脱敏图纸、学校、企业、学生信息 | 提醒脱敏处理 |

## Docker本地验证期望

- `docker compose up` 后控制台可访问。
- 健康检查能显示服务、知识库、模型配置状态。
- 至少完成一次普通问答 smoke test。
- 至少完成T01-T08测试入口的读取和一条真实测试执行。
- 最终验收仍以T01-T08全部真实模型测试和人工判定为准。
